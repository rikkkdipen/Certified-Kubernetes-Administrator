PODS (Kubernetes / CKA Notes)
============================

1) What is a Pod?
-----------------
- A Pod is the smallest deployable unit in Kubernetes.
- It wraps one or more containers that must run together.
- Containers in the same Pod share:
  - the same network namespace (one Pod IP)
  - localhost communication
  - shared volumes (if configured)

2) Why Pods exist (not just containers)?
---------------------------------------
- Kubernetes schedules Pods (not individual containers).
- A Pod represents one “application instance”.
- Sidecars run in the same Pod (e.g., logging agent, proxy) when tight coupling is needed.

3) Pod networking (must know)
-----------------------------
- Each Pod gets a unique IP inside the cluster.
- All containers in the Pod share that IP.
- Ports are per container, but reachable via PodIP:Port.
- Pod-to-Pod networking is handled by the CNI plugin.

4) Pod lifecycle & states (CKA focus)
-------------------------------------
Common Pod statuses (kubectl get pods):
- Pending:
  - not scheduled yet OR waiting for image pull/volume/network sandbox
- Running:
  - at least one container running and none failed (but may not be Ready)
- Succeeded:
  - all containers completed successfully (common for Jobs)
- Failed:
  - all containers terminated and at least one failed
- Unknown:
  - kubelet can’t be reached

Important container states inside describe:
- Waiting:
  - ImagePullBackOff / ErrImagePull / ContainerCreating
- Running
- Terminated:
  - Error / Completed / OOMKilled

5) Pod readiness vs running
---------------------------
- "Running" does not always mean "Ready".
- Readiness is controlled by readinessProbe.
- Services send traffic only to Ready pods (endpoints are built from Ready pods).

6) What kubectl describe pod tells you (debugging gold)
-------------------------------------------------------
- Node chosen (scheduler output)
- Pod IP
- Container image, image ID (digest), container ID
- State + Reason (Waiting/Running/Terminated)
- Restart count
- Mounts (service account token, volumes)
- Conditions (PodScheduled, Initialized, Ready, ContainersReady)
- Events (most important for root cause)

7) Labels & selectors (service routing)
---------------------------------------
- Labels are key/value tags on Pods.
- Services select pods by labels.
- If selector doesn't match labels -> endpoints empty -> service doesn’t work.

8) Probes (health checks)
-------------------------
- startupProbe: for slow-starting apps
- readinessProbe: controls if pod receives traffic
- livenessProbe: restarts container if unhealthy

9) Resource requests/limits + QoS
---------------------------------
- Requests affect scheduling decisions.
- Limits cap usage.
QoS classes:
- BestEffort: no requests/limits (first to be evicted)
- Burstable: some requests/limits
- Guaranteed: requests == limits for all containers

10) Common CKA pod issues & quick direction
-------------------------------------------
- Pending + FailedScheduling: scheduler/taints/resources/affinity
- ImagePullBackOff: registry/network/auth/image tag issues
- CrashLoopBackOff: app crash; check logs and config
- ContainerCreating: CNI/network or volume mount delays
- FailedMount: Secret/ConfigMap/PVC missing or mount error
- NotReady: readiness probe failing

11) Practical access methods
----------------------------
- Exec into pod:
  kubectl exec -it <pod> -- sh
- Logs:
  kubectl logs <pod>
  kubectl logs <pod> --previous
- Port-forward:
  kubectl port-forward pod/<pod> 8080:<container-port>
- Service access:
  kubectl expose pod <pod> --port=80 --name=<svc>
  kubectl port-forward svc/<svc> 8080:80

12) One-liner to remember
-------------------------
- Pods are the unit Kubernetes schedules and runs.
- Debug pods using: describe + events + logs.
